**salesforce** プラグインはSalesforce CLIをClaude Codeと統合し、組織認証とSOQLクエリを含む任意の `sf` コマンドを実行できる汎用CLIエージェントを提供します。JWT、アクセストークン、SFDX URL認証フローによるヘッドレスコンテナ環境をサポートしています。以下の使用ガイドでは、CLIエージェントを通じて自然言語でアカウント、商談、ケース、取引先責任者をクエリする方法を説明します。

<Badge text="v1.0.0" variant="note" /> <Badge text="Development" variant="tip" />

## インストール

```
/plugin install salesforce@f5xc-salesdemos-marketplace
```

## 前提条件

- **Salesforce CLI** (`@salesforce/cli`)
- APIアクセスが有効な**Salesforce組織**
- オブジェクト（ケース、商談、取引先、取引先責任者）をクエリする権限を持つユーザーアカウント

ワークステーションにSalesforce CLIをインストールします：

```bash
brew install sf
```

インストールを確認します：

```bash
sf --version
```

`@salesforce/cli/2.x.x` のような出力が表示されるはずです。

## 認証セットアップ

### ステップ1：Salesforceドメインを確認する

ブラウザでSalesforceのURLを確認してください。Salesforceに
`https://acme.lightning.force.com` でアクセスしている場合、ログインドメインは
`https://acme.my.salesforce.com` です。

<Aside type="tip">
  会社がSSOを使用している場合、`--instance-url` でカスタムドメインを指定する必要があります。
  これにより、一般的なSalesforceログインページではなく、SSOプロバイダーを通じてリダイレクトされます。
</Aside>

### ステップ2：ブラウザ経由で認証する

```bash
sf org login web --alias my-org --set-default --instance-url https://YOUR-DOMAIN.my.salesforce.com
```

`YOUR-DOMAIN` をステップ1で確認した会社のドメインに置き換えてください。ブラウザが開くので、SSOまたはログインフローを完了し、アプリを承認してください。以下のように表示されるはずです：

```
Successfully authorized your-email@company.com with org ID 00DXXXXXXXXXXXXXXX
```

### ステップ3：接続を確認する

```bash
sf org display --target-org my-org
```

出力にユーザー名とインスタンスURLとともに `Connected Status: Connected` が表示されることを確認してください。

### ステップ4：コンテナ向けポータブル認証（オプション）

ヘッドレス環境（CI/CD、コンテナ、リモートセッション）で認証する必要がある場合、ワークステーションからSFDX認証URLをエクスポートします：

```bash
sf org display --verbose --target-org my-org
```

**Sfdx Auth Url** の値（`force://` で始まる）をコピーします。コンテナ内で以下を実行します：

```bash
export SFDX_AUTH_URL="force://PlatformCLI::YOUR_AUTH_TOKEN@your-domain.my.salesforce.com"
echo "$SFDX_AUTH_URL" | sf org login sfdx-url --sfdx-url-stdin=- --alias=my-org --set-default
```

<Aside type="caution">
  SFDX認証URLは認証情報です — パスワードと同様に扱ってください。ファイルにコミットしたり、
  チャットに貼り付けたりしないでください。環境変数とstdinパイプのみを使用してください。
</Aside>

### 認証方法リファレンス

| 方法 | 最適な用途 | 必要なもの |
|--------|----------|----------|
| Webログイン | ブラウザのあるワークステーション | ブラウザ + SSO |
| SFDX URL | コンテナ、CI/CD、ポータブル認証 | 認証済みセッションからの認証URL |
| JWTベアラー | 自動化パイプライン | 接続アプリ + 秘密鍵 + コンシューマーキー |
| アクセストークン | 環境変数認証 | `SF_ACCESS_TOKEN` + `SF_ORG_INSTANCE_URL` |

**注意：** デバイスフロー（`sf org login device`）は2025年8月以降ブロックされており、サポートされていません。

### 環境変数

| 変数 | 用途 |
|----------|---------|
| `SF_ACCESS_TOKEN` | アクセストークン認証用のベアラートークン |
| `SFDX_AUTH_URL` | SFDX URL認証用のForce認証URL |
| `SF_ORG_INSTANCE_URL` | 組織のインスタンスURL |
| `SF_JWT_KEY_FILE` | JWT秘密鍵のパス |
| `SF_CLIENT_ID` | 接続アプリのコンシューマーキー |
| `SF_USERNAME` | JWT用のSalesforceユーザー名 |

## コマンド

### /salesforce:sf-status

Salesforce組織の接続状態、CLIバージョン、認証済み組織を確認します。

```
/salesforce:sf-status
```

**期待される結果：** sf CLIバージョン、認証済み組織のエイリアス、ユーザー名、インスタンスURL、接続状態、APIバージョンを示すテーブルが表示されます。

### /salesforce:sf-login

Salesforce組織に認証します。オプションでエイリアスを指定できます。

```
/salesforce:sf-login
/salesforce:sf-login my-org
```

**期待される結果：** そのエイリアスの組織が既に認証済みの場合、接続の詳細が報告されます。認証されていない場合、利用可能な認証情報（環境変数）を確認し、最初に条件を満たした方法で認証します。

## スキル

### salesforce-index

トップレベルのインテントルーター。自然言語でSalesforce、組織管理、またはデータクエリに言及すると自動的にアクティブになります。

| ユーザーの発言 | ルーティング先 |
|---------|-----------|
| "salesforceにログイン"、"組織を認証" | `salesforce-auth` スキル |
| "組織のステータスを確認"、"組織を一覧表示" | `cli-operator` エージェント |
| "商談を表示"、"ケースを検索" | `cli-operator` エージェント（SOQLクエリ） |
| "Apexクラスを生成"、"Flowを作成" | afv-library 開発スキル |

### salesforce-auth

コンテナ対応の認証スキル。4つの認証方法をサポートし、利用可能な認証情報を自動的に検出します。優先順位：アクセストークン、JWT、SFDX URL、Webログインの順です。

## エージェント

### cli-operator

安全ガードレール付きで `sf` コマンドを実行する自律型Salesforce CLIエージェント。メインセッションのコンテキストを軽量に保つため、すべてのSalesforce操作はこのエージェントに委任されます。

**安全ルール：**

- デフォルトで読み取り専用 — 書き込み操作には明示的な承認が必要
- プレビューと確認ステップなしにデプロイしない
- アクセストークン、認証URL、認証情報を出力にエコーしない
- ユーザー提供の値をすべてサニタイズ（シェルのメタ文字を拒否）
- 構造化された解析可能な結果のために `--json` 出力を使用

**レスポンス形式：**

```
## Result: [SUCCESS | FAILURE | PARTIAL]
### Command Executed
### Output Summary
### Issues
```

## 使用ガイド：アカウント管理

<Aside type="note">
  以下のプロンプトはプラグインのハードコードされた機能ではありません。cli-operatorエージェントが
  任意のSOQLクエリを含む任意の `sf` CLIコマンドを実行できるため機能します。結果は組織のデータモデルと
  ユーザー権限に依存します。ここに記載されたプロンプトは標準Salesforceオブジェクト
  （Account、Contact、Case、Opportunity、AccountTeamMember、OpportunityTeamMember）でテストおよび検証済みです。
</Aside>

これらのプロンプトはどのSalesforce組織でも機能します。プレースホルダーの値を自分の情報に置き換えてください。各プロンプトは、あなたのロール、アカウント、テリトリーに固有の結果を生成します。

<Aside type="tip">
  最良の結果を得るには、ユーザーレコードの特定が必要なプロンプトにメールアドレスを含めてください。
  Salesforce組織には類似した名前の複数のユーザーが存在する場合があります。
</Aside>

### はじめに

認証後、すべてが正常に動作することを確認します：

```
/salesforce:sf-status
```

次に、簡単な自然言語クエリを試してみましょう：

```
list my authenticated salesforce orgs
```

**期待される結果：** エイリアス、ユーザー名、インスタンスURL、接続状態を含むすべての接続済み組織を示すテーブルが表示されます。

### アカウントの検索

自分がどのアカウントに割り当てられているかを確認します：

```
what salesforce accounts am I on the account team for? My email is your-email@company.com
```

**期待される結果：** アカウントオーナーごとにグループ化されたアカウントの一覧と、各アカウントでのチームメンバーロールが表示されます。ほとんどのロールが空白の場合、これはよくあるデータ品質の問題です — アカウントチームメンバーシップは存在しますが、ロールが設定されていません。

### 同僚とのカバレッジの相互参照

チームメイトとのアカウントカバレッジを比較してギャップを見つけます：

```
In salesforce, find all accounts where Colleague Name is on the account team. Then for each of those accounts, check if your-email@company.com is also on the account team. Show me two lists: accounts where we are BOTH tagged, and accounts where only my colleague is tagged but I am missing.
```

**期待される結果：** 重複数とギャップ数を示すサマリーテーブルの後に、2つのリストが表示されます。最近ロールを変更した場合、重複がゼロになることがあります — これにより、自分が追加される必要のあるアカウントが明らかになります。

<Aside type="note">
  前任者の確認も追加できます：「Also check if
  Previous Person Name appears on any of those accounts.」これにより、
  テリトリーで誰かの後任になる際の引き継ぎギャップを特定できます。
</Aside>

### テリトリーパイプライン概要

アカウント全体のオープン商談を確認します：

```
show me all open salesforce opportunities on Colleague Name's account team accounts, sorted by close date, include the stage, amount, and probability
```

**期待される結果：** 商談の総数、パイプラインの生値、加重パイプラインを含むサマリーが表示されます。その後、ステージ分布テーブルと金額順の上位商談リストが続きます。過去の完了予定日やnullの金額などのデータ品質の問題がフラグ付けされます。

### 商談の詳細分析

特定の案件の全体像を把握します：

```
show me a detailed view of the OPPORTUNITY NAME opportunity in salesforce - include the opportunity team members, any activities or tasks, and the account contacts
```

**期待される結果：** 商談の概要（ステージ、金額、完了予定日、確度、予測カテゴリ、オーナー）、ロール付き商談チームメンバー、タグ付けされた取引先責任者、最近のタスクとイベント、案件金額と完了予定日の推移を示すフィールド変更履歴が表示されます。

### テリトリー全体のサポートケース

アカウントのオープンケースを確認します：

```
show me all open salesforce cases across Colleague Name's account team accounts, grouped by account, sorted by most recent first
```

**期待される結果：** 優先度とステータスの内訳を含むオープンケースの件数、その後アカウントごとにグループ化されたケースの詳細が表示されます。活動のない古いケース（数か月または数年）はデータ品質改善の候補としてフラグ付けされます。

### 四半期パイプライン予測

パイプラインの予測ビューを取得します：

```
for Colleague Name's accounts, show me a quarterly pipeline summary - group the open opportunities by close date quarter with count, total amount, and weighted amount for each quarter
```

**期待される結果：** 四半期ごとに商談数、総パイプライン、加重パイプラインを示す1行のテーブルが表示されます。四半期ごとのステージミックスの内訳と上位の加重案件がハイライトされます。過去の完了予定日の商談は別グループにまとめられます。

### アカウント全体概要

特定の顧客を詳しく分析します：

```
give me a full account overview for ACCOUNT NAME in salesforce including contacts, open opportunities, and recent cases
```

**期待される結果：** 企業プロフィール（業種、売上、従業員数、所在地）、役職とメール付きの主要な取引先責任者、ステージと金額付きのオープン商談、ステータス付きの最近のサポートケースが表示されます。

### ケース検索

番号で特定のサポートケースを検索します：

```
look up salesforce case CASE-NUMBER and show me the case details, the customer account, and who owns it
```

**期待される結果：** ケースの件名、ステータス、優先度、説明、顧客アカウントプロフィール、連絡先情報付きのケースオーナーが表示されます。

## 使用ガイド：データクエリ

パワーユーザー向けに、自然言語で特定のSOQLスタイルのクエリをリクエストできます。プラグインがリクエストを適切なSOQLに変換して実行します。

### 特定のフィールド

```
query salesforce for all Contacts at ACCOUNT NAME - show Name, Title, Email, Phone, and Department
```

### 日付フィルタリング

```
show me all salesforce opportunities that closed won in the last 90 days on ACCOUNT NAME
```

### 集計

```
count all open salesforce cases grouped by priority and status across my accounts
```

### カスタムオブジェクト

```
query the CUSTOM_OBJECT__c object in salesforce for records where Status__c = 'Active'
```

<Aside type="tip">
  フィールドが存在しないためにクエリが失敗した場合、プラグインは自動的にそのフィールドを除外して
  再試行します。組織のカスタムフィールドは異なる名前を使用している場合があります —
  フィールドが見つからない場合はSalesforceの設定を確認してください。
</Aside>
